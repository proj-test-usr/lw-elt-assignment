# Assignment – dbt Project

## Overview

This dbt project builds a data pipeline that calculates Monthly Recurring Revenue (MRR) source locally from csv files (main_pipeline\seeds).

The final output is a mart table that attributes revenue to:
- School use case
- Customer country
- Calendar month

Revenue is amortized across billing periods (monthly, quarterly, annual), and credit notes are handled as negative revenue.

---

## Project Architecture

The project follows a standard dbt layered approach:

sources → staging → intermediate → marts

ensuring modular transformations, maintainability, testability, clear lineage and scalability.

## Final Mart

The final export is generated by a python script (\scripts) extracting the following table from the local duckdb. Rounding of the mrr_usd is applied during export for visualization purposes; actual table is stored with precision.

### fct_mrr_by_school_use_case_country_month

**Grain:**
- one row per month
- per use case
- per country

**Columns:**
- month
- use_case
- country
- mrr_usd

## Testing Strategy

### Generic tests
- not_null on keys and dimensions
- uniqueness on primary keys
- accepted values tests on products, subscriptions, customers
- unique constraint on mart grain
- relationships between facts and dimensions

### Custom tests
- Revenue preservation: total amortized revenue must equal total invoiced revenue.
- no doublicate months, ensuring invoice-month allocation is unique (no double counting)
- valid billing date ranges
- non-null and numeric revenue
- integrity of amortized revenue

---

## Key Assumptions

- Revenue is distributed evenly across billing periods (per requirement)
- Partial periods are included in full month allocations (month vs. day granularity)
- Credit notes are treated as negative revenue and amortized over their covered billing period
